# アプリケーション設計

ソフトウェアの構造やシステムの振る舞いを計画し、**どのようにして要件を満たすかを決定するプロセス**。

設計では「機能要件（何をするか）」「非機能要件（パフォーマンス・拡張性・セキュリティ）」「システム構成（モジュール分割）」「データ・処理の流れ」を定義する。

## ソフトウェア設計の基本概念

### カプセル化
データと操作をオブジェクトにまとめ、内部状態を外部から直接アクセスできないよう隠蔽する。`private` 属性に対して `public` メソッド経由でのみ操作させる設計。

### 抽象化
本質的な要素や共通の振る舞いを抽出し、不要な詳細を排除する。複雑さを管理し、再利用性を高める。

### インターフェース
クラスが実装すべきメソッドのセットを定義する契約。実装の強制・抽象化の促進・依存関係の削減が目的。

## モジュール設計の基礎

**モジュール** = プログラムを機能ごとに分割した独立性の高い構成部品（クラス・パッケージ・名前空間など）。

設計で意識すべき3つのポイント：

| ポイント | 概要 |
|---------|------|
| **単一責任の原則 (SRP)** | モジュールはたった一つのアクターに対して責務を負う |
| **凝集性** | モジュール内部の機能は強く関連したものだけをまとめる（高凝集を目指す） |
| **関心の分離 (SoC)** | UI・ビジネスロジック・データアクセスなど、異なる関心事を独立したモジュールに分ける |

---

## モジュール間の関係性

依存の方向性は **一方向依存（理想）→ 双方向依存（避ける）→ 循環依存（最も避ける）** の順で問題が大きくなる。

**結合度** はモジュール間の依存の強さを示す指標。密結合を避け、疎結合を目指す。

依存関係を管理する3つのテクニック：

| テクニック | 概要 |
|-----------|------|
| **インターフェースの活用** | 具体的な実装ではなく抽象（インターフェース）に依存させる |
| **依存性逆転の原則 (DIP)** | 上位・下位モジュールともに抽象に依存させ、具体への直接依存をなくす |
| **Dependency Injection (DI)** | 依存オブジェクトを外部から注入し、クラス内部での生成をなくす |

## アーキテクチャの基礎

**アーキテクチャ** = システムの全体構造・各部品の役割・依存関係・データの流れ・守るべきルールを定義する設計図。

アーキテクチャの種類：
- **システムアーキテクチャ** — 複数サービスの全体構成
- **インフラアーキテクチャ** — サーバー・ストレージ・ネットワークの設計
- **アプリケーションアーキテクチャ** — アプリ内部の構造とデータの流れ（本リポジトリの学習対象）

主なアプリケーションアーキテクチャパターン：

| パターン | 特徴 | 使い所 |
|---------|------|--------|
| **レイヤードアーキテクチャ** | 層ごとに責任分担、下の層にのみ依存 | シンプルで理解しやすい、多くのWebアプリに適用可能 |
| **クリーンアーキテクチャ** | ビジネスロジックを中心に円構造、依存はすべて内側向き | 長期保守が必要な大規模・複雑なアプリ |

アーキテクチャ選定の観点：プロジェクト規模・寿命・チームの技術力・ビジネスドメインの複雑さ。
